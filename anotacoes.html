ANOTAÇÕES SOBRE O PROJETO

**TAG NOVA OVERFLOW: Essa tag faz com que qualquer conteúdo que esteja sendo "vazado" de caixas filhas, seja ESCONDIDO. Ou seja, imagens ou qualquer outra coisa que ultrapasse os limites da caixa mãe serão ocultados.

****SOBRE COMO AGIR COM O FLOAT (Coisas em que eu tive dúvidas): No media query para tablet, o projeto pediu que as duas divs (de imagem e formulário) agora ficassem uma ao lado da outra. Foi possível denominando que a div imagem fosse um "float: left" e a div do formulário fosse um "float: right". Só que a div da imagem desapareceu pois o width estava zerado (por algum motivo que eu desconheço). Logo, resolvíamos colocando um valor para o width da imagem. Mas, nas tentativas sozinho, outro fator que fazia com que as divs "bugassem" ou "sumissem" era que, dado um valor de width para cada uma das divs, se esse valor SOMADO fosse MAIOR do que a largura (width) da container mãe (que é a section de id login), as divs se EMPILHAVAM, dando o efeito de que tinham sumido. Logo, não adiantava muito colocar valores específicos em píxel para as duas divs, pois conforme eu ia testando a responsividade (esticando ou diminuindo a tela), o valor de width da section que é a container mãe mudava, fazendo com que ou as divs ficassem muito longe uma das outras (no caso de esticar), ou que a soma das duas ficasse maior do que ela (no caso de encolher a tela).
ENTÃO, a solução foi colocar valores de width para as divs em PORCENTAGEM, para que ficassem sempre proporcionais ao valor da section container mãe!
(Outra coisa que não entrava na minha cabeça de jeito nenhum era o porquê de a div formulário não conseguir ocupar a altura inteira da section, era pq o form no HTML estava vazio e estava empacando o resto do conteúdo de crescer)

EFEITO TRANSITION: O efeito transitio do projeto foi feito usando realmente a tag transition, PORÉM, COM VALORESSSSSSS DE WIDTH E HEIGHT, com os tempos e separados por vírgula. Assim: transition: width 0.3s, height 0.3s;

SUAVIZADA NO TRANSITION: Usa-se a propriedade chamada transition-timing-function: ease; para dar um efeito de suavidade na animação do transition!

******DICA IMPORTANTE: Para no CSS sinalizarmos um elemento pelo seu TIPO, colocamos a tag normalmente e, ao invés de usar ponto para classe ou hashtag para ID, usa-se colchetes e escreve-se o type como se fosse no HTML (EXEMPLO NO PROJETO, ONDE SINALIZAMOS NO CSS, SEM USAR CLASS, O INPUT SUBMIT PELO TYPE DELE, QUE É TYPE=SUBMIT).
Assim: input[type=submit] {}

*********MUITISSIMO IMPORTANTE: DIMINUINDO O TAMANHO DOS ÍCONES COLOCADOS NO PROJETO: Tentei de várias formas diminuir aqueles ícones que ficavam acima da linha da palavra mas não conseguia. O JEITO PARA MUDAR O TAMANHO DO ÍCONE É USANDO FONT-SIZE!!! MAS, NÃO FUNCIONOU DE FORMA ALGUMA TENTANDO COLOCAR CLASS OU ID COM UM FONT-SIZE PARA ELE, E EU NÃO SEI O PORQUÊ, MAS O ÚNICO JEITO FOI COLOCANDO O CAMINHO TODO ATÉ CHEGAR NO ELEMENTO QUE CONTINHA O ÍCONE!!! (NO CASO, UM SPAN)
Assim: form > a.botao > span {font-size: 0.8em;}

PARA MUDAR A COR, FOI PRATICAMENTE A MESMA COISA!! Só que, logicamente, usa-se o "color" que funcionou para trocar a cor dos ícones dos campos de login!!! (Dessa vez, nos ícones dos campos de login, testei colocar uma class para mudar a cor e funcionou, o que não tinha acontecido com o ícone do mail anterior, ou talvez porque era font-size...)
***ATENÇÃO: De qualquer jeito, se atentar MUITO com esses ícones, e sempre lembrar de USAR SEMPRE O CAMINHO para quando formos definir algo sobre eles.

****"CENTRALIZANDO" OS ÍCONES:Por padrão quando adicionamos os ícones, eles ficam meio para cima em relação ao conteúdo (coisa que eu não consegui mexer de jeito nenhum sozinho). Pelo que eu entendi até agora, podemos "centralizar" o ícone primeiramente nos referindo a eles no CSS com o CAMINHO QUASE TODO praticamente (como dito acima) e, depois, aumentando um pouco o tamanho dele usando font-size, aumentando um pouco a caixa dele usando width, e depois APLICANDO UM PADDING PRA ELE DAR UMA DESLOCADINHA PRO MEIO!!

***AJEITANDO O TAMANHO DO INPUT DE LOGIN: Aparentemente, o font-size também serve para aumentar o tamanho do input (verticalmente, como se fosse um height), mas não serve para determinar o tamanho real de configuração que queremos.

******ALGO MUITO IMPORTANTE A SER LEVADO EM CONSIDERAÇÃO: Sobre centralizar e/ou ajeitar um campo ou objeto, eu não sei como será daqui para frente com os estudos das tecnologias de grid, flexbox etc. Mas, uma coisa que (pelo menos até agora) eu aprendi, é que nem sempre (pelo menos até agora sem as tecnologias novas citadas) as coisas ficarão centralizadinhas ou certinha da forma que queremos. E para isso, às vezes é preciso recorrer a artimanhas e "pequenos píxels" para conseguirmos colocar algo onde queremos, EXEMPLO: A CAIXA DE LOGIN DO PROJETO LOGIN, onde mesmo tentando fazer com que ambos o ícone e o campo de login tivessem um width onde juntos somados dava 100% para que fossem os dois encaixados na caixa (tentei fazer os ícones com width de 10% e a caixa de login 90%), não dava certo e a caixa de login "QUEBRAVA" para a diagonal, ou seja, não ficava certinha nem à direita e nem para cima.
Como foi feito então:

  1. width: calc(100% - 45px); ENCAIXANDO O CAMPO DE LOGIN HORIZONTALMENTE: Sim, pode-se calcular no CSS para definir com mais precisão a diferença entre dois elementos em uma caixa (e ainda pode-se calcular porcentagem com píxel). No caso, a intenção era fazer o campo de login ocupar a maior parte e esse cálculo foi feito para atribuir à caixa de login o width mais preciso, já que o outro elemento tinha width de 40px (sim, não foi exatamente o valor definido no cálculo, mas foi a melhor diferença encontrada, pois mesmo usando o exato valor de width do ícone, o campo de login ainda "quebrava"), a diferença entre eles então seria o width da caixa de login! Tendo então feito isso, ainda sobrava um mínimo píxel de diferença entre o encaixe do campo de login, com a div container atrás, mas ainda assim foi o mais próximo encontrado. Foi resolvido colocando uma borda da mesma cor do background, para dar um efeito de bordado mesmo (criatividade).
  2. transform: translateY(-10px); ENCAIXANDO O CAMPO DE LOGIN VERTICALMENTE: Tendo então conseguido "encostar" o campo de login horizontalmente no limite onde queríamos, faltava encaixar verticalmente. Já visto antes, o transform translate nos permite reposicionar certo elemento, lembrando que o valor negativo faz com que o elemento ande para o lado inverso, pois o padrão é sempre para baixo e para direita, já que o ponto de ancoragem é sempre o ponto superior esquerdo (nesse caso para CIMA, pois o valor usado é o "translateY", que é apenas no eixo VERTICAL)
    NOTA: transform: translate sem Y ou X definido é quase um "shorthand" onde aceita os dois valores (como já visto na centralização de algum objeto feito anteriormente).
    No caso então: transform: translate(eixo X, eixo Y)
    NOTA 2: Quando tentei sozinho, tentei usar a propriedade bottom para tentar posicionar a caixa de login verticalmente e até que consegui, mas só funcionou na versão mobile, nos outros media queries ficou desencaixado. Logo, o translate funcionou para também as outras media queries, e o bottom só funcionou para o mobile.

DICA: O professor disse que todas as propriedades de transition, transform e translate devem ficar sempre em ÚLTIMO na declaração de CSS, pois podem haver bugs devido a elas!

PSEUDO CLASSE "FOCUS-WITHIN": É uma pseudo classe muito legal que dá aquele efeito de, onde o campo estiver selecionado, a propriedade que colocarmos será mostrada (Um exemplo bem comum, é quando um campo está atualmente selecionado, ele fica branco, poi tem justamente um background color white, como também foi usado no projeto).
